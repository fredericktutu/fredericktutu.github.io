(软件开发)我想成为架构师

post @ 2020-04-25

我对于自己的职业规划一直是：成为一个编译工程师+软件架构师(全栈工程师)。这一开始是我自己的美好展望，不过随着我看的书越来越多，也开始做越来越多的项目，我首先开始觉得第一个目标有机会实现，不过第二个目标却好像遥不可及。

原因有很多。首先，我没有系统性的学习过软件的开发；其次我有过两三次失败的项目经历，可以说是一个没有“冠军基因”徽章的球员。

而最近，我在一个平台开发的项目中担任leader（虽然我觉得我并不算很称职），可以说感触良多。虽然不知道我现在做的事情，是不是类似于软件架构师，我想“提前”总结一下经验。这些经验正不正确，让未来的我来评价吧。

架构师的技术

引子-秦始皇的计算机

之前在知乎上看到一个文章，用故事的方法，讲了一个故事：程序员“丞相”和秦始皇站在高高的指挥台上，丞相演示了一个特殊的“计算机”。这里的计算机，是由亿万个士兵组成的，他们有的代表内存，每一个人代表一个比特位；有的代表运算器；有的代表数据通路。他们一起形成了一个宏大壮观的计算机。

讲这个故事是为了描述我目前对于一个架构师的理解。我以前总以为架构师做的是很宏观的事情，比如：

  • 1.划分模块.把一个图书馆分为1前端、2控制器、3后端逻辑，blabla
  • 2.定义框架，定义设计模式. 比如整个架构符合MVC模式，我们采用Spring框架，blabla
  • 3.各个小组分工开始干活吧

这样的说法，也对也不对。为什么？

首先，一个架构师当然需要做宏观的事情，比如上面说的模块的设计和划分。但是如果架构师只做了这么一件事情，那么最后的项目一定是失败的。因为框架和设计模式，它们是解决问题的工具，但不是分析问题的工具。如果一个架构师一开始就开始考虑框架和设计模式，要么他是一个非常有经验的架构师，但更多情况，他和我一样是一个菜鸟架构师。

《软件工程》中的方法真的没用吗？

上面我们提到了分析问题，那么如何分析问题呢？没错，用的就是各大《软件工程》中提倡的面向对象方法。但是，从我在社交媒体上看到的情况来说，大多数同学并没有理解这种分析方法，因此，我想聊一聊面向对象分析。

这里讨论需求分析和详细设计，前者对应分析，后者对应设计。

首先，对于需求分析，有人自以为是地将其总结为“套公式”(套类图、套CRC卡片、套动态模型)。其实相去甚远，需求分析做的事情，是模拟,并且模拟的很通透！

打个比方，对于一个自动售卖机，我们怎么模拟？

我们会想象一个自动售卖机里有若干个槽位装着零食，然后我们会想象售卖机有一个营业员,有一个跑腿员,一个记录员。当我们买东西时，我们给营业员钱，告诉营业员我要买什么，营业员询问记录员零食还有没有，得到肯定答复以后，他通知跑腿员去把零食取出槽位并给用户。

我们可以看到，上述过程有以下好处：

  • 对于以上的需求分析，过程是极为具体的；
  • 同时，分析它的过程也是非常愉快的，因为我们用了拟人的手法233（据我的经验表明，拟人真的是非常好的一种方法，我可能会在后面的博客中专门讲一讲它）。
  • 还有一个重点，就是在做完这个分析以后，我们已经有8成的把握完成它了。
    这些结论可以说显而易见，不需要再证明了把。

而之后的详细设计，便是设计模式和框架发挥作用的时候了，这个时候，你可以把会的东西全都拿出来，好好的展示一下你的技能。

面向对象分析的两种误区

第一种：“需求分析不用涉及到具体实现，只需要大概地描述功能就可以了”。

这一类就是上面说的“套公式”，他们拿到任务，便开始在运用在OOP课程上学的的些许知识和自己的常识定义一些类，然后完成类图、CRC等。通常来说，对系统怎么运作起来的，他们很难给出一个自洽的解释。

一开始，我其实对他们的话深信不疑，也认为需求分析不需要做的这么具体。然而，当我一次次发现那些类的定义漏洞百出，根本无法自洽时，我不再相信“套公式”产生的需求分析。

第二种：“我在类图的结构中使用了MVC的架构，可以实现高内聚低耦合”

这一类说的就是我自己，当我看了《head first 设计模式》后，我对设计模式的力量震撼到了，并非常想让它融入进我的项目里，但是这样有点太急躁了，可以看看我第一次根据设计模式做出的需求分析类图：

再看看我根据更好的方法分析出的类图：

可以看到，后一个更容易让人理解。说实话，第一个，我在和组员讲解的时候都缺乏底气。

设计模式无处不在

上一节批判了我自己乱用设计模式，但是这一节，我想为设计模式正名，展现他在设计中的强大威力。

(编译原理)国外大学编译相关课程整理

post @ 2020-04-16

Intro

这里简单整理了一下外国大学的一些编译相关的课程，主要分为以下几类

  • 介绍Parsing系统，以及编程语言特性的实现 CS164(C++ 写Python) CS2111
  • 主要讲编程语言泛型 CS107 CS422
  • 讲现代编程语言的运行时的架构 CS263

我的计划

我目前想先学习CS263，了解现代编译器的一些技术；接下来想学习CS422，主要是想再深入挖掘PL中的一些概念。

CS164: Programmming Language and Compilers

  • CS164 is an introduction to the design of programming languages and the implementation of translators for them. In the process, we’ll do some general exploration of programming language design and its implications for implementation, and look at a dialect of at least one particular language, which this semester is Python.

CS2111： Design and Implementation of Programming Languages

  • 1999年的老课程了，其中的实验可能现在比较难做
  • This course unit examines programming languages, and in particular imperative languages, looking at the fundamental concepts they have in common, and outlining how to implement them. We also look at the variations in these concepts between languages, and see how this affects usability and ease of implementation. We will investigate these concepts using examples from various languages, particularly C and SML, rather than examining complete languages in turn.

Stanford CS107: Programming Paradigms

  • Advanced memory management features of C and C++; the differences between imperative and object-oriented paradigms. The functional paradigm (using LISP) and concurrent programming (using C and C++). Brief survey of other modern languages such as Python, Objective C, and C#.

  • Prerequisites: Programming and problem solving at the Programming Abstractions level. Prospective students should know a reasonable amount of C++. You should be comfortable with arrays, pointers, references, classes, methods, dynamic memory allocation, recursion, linked lists, binary search trees, hashing, iterators, and function pointers. You should be able to write well-decomposed, easy-to-understand code, and understand the value that comes with good variable names, short function and method implementations, and thoughtful, articulate comments.

UIUC CS422 Programming Language Design

  • major language features including: an understanding of static versus dynamic binding, call-by-value, call-by-reference, call-by-name, and call-by-need, type checking and type inference, objects and classes, concurrency;
  • formal methods of specifying the semantics of programming languages including: Structural Operational Semantics, Transitions Semantics, Chemical Abstract Machine, K;
  • One goal of this course is to explore the structure of programming languages and to consider alternatives to familiar programming language features. We’ll also study the problem of translating programming languages into machine-executable forms, using Python as a concrete example of a language to be translated, and the assembly language of the Intel ia32 family (used in PCs and some of our Solaris workstations downstairs) as a concrete example of a target machine. We study language translation first to learn some of techniques used that are useful for many programming problems outside of language translation, second to gain a better intuitive feel for the tools we use when programming and the costs of the programs we write, and third (possibly most important) to gain experience with the engineering problems associated with building and validating a substantial piece of software.

CS263: Programming Language Runtime Systems

  • 讲的是编程语言运行时的一些东西，比如说JVM JIT等

(阿里实习面试)三轮技术面试以及HR面试的总结

post @ 2020-04-12

以下是我在面试阿里云基础软件事业部编译器组的过程，面试期间共经过了三轮技术面试，一轮HR面试. 整个流程从3月18日开始，到4月10日左右，目前正在等待阿里的通知,希望有好运!

2020/4/25 这里更新一下，上周已经被录取啦，接下来学习C++准备去上班

关于简历

我的简历主要写了以下模块：

  • 教育经历：这一部分写了我在武大的学习成绩和核心课程的情况，以及我在加州大学伯克利分校参加暑期课程的情况（这段经历主要是为了说明我对编译器感兴趣的原因）
  • 专业技能：这一部分主要写了我所会的编程语言，我所使用过的软件框架，我会使用的工具，我的英语水平。
  • 项目经历：主要写了三个项目lisp解释器,对象代理数据库解释器，微信小程序

关于自我介绍

自我介绍非常重要，能够让面试官了解自己的技术栈，从而定制化地问你一些问题，不然可能会多问一些基础问题。

在每一轮我的自我介绍都差不多，include了以下内容：

  • 来自武汉大学弘毅学堂计算机方向
  • 感兴趣的技术方向compiler & Parallelism & NLP
  • 主要做过的项目，这一部分我尽可能地把我参与的项目都说了一遍，分成三个部分
    • 编译器： Lisp解释器with SVM（个人项目）， 对象代理数据库解释器（数据库实践），Python子集解释器（软工项目）
    • 硬件： Mips 32 单周期/多周期/流水线/中断 CPU的实现
    • 软件： 微信小程序/ 管理系统

第一轮面试（同事面）

这一轮面试是由未来的同事对你进行面试，考察的内容比较具体，偏重于基础知识和算法，大概的内容如下：

问项目：关于Lisp解释器的虚拟机

面试官对我自己做的这个项目比较感兴趣，问了我虚拟机底层的代码形式是怎么样的。我结合了一个具体的Lisp程序，解释了生成的底层代码的样子。同时我还解释了我在执行上的一个逻辑，特别讲了我自创的”mode”和”method”的定义。

然后面试官问如果需要访问非栈顶元素，怎么办呢？那我的回答是，对于内置函数，底层的代码实际上是可以访问到非栈顶元素的，但面试官好像对这个答案不是特别满意，然后就没有继续聊这个项目了。

今天我想了想，可能面试官想问的是对于用户定义的函数，如何访问到某个具体的形参。这确实是我理解的问题，如果当时能理解，本来可以继续再聊一聊关于嵌套的动态作用域的问题233. 不过总体来说，开始聊的比较开心。

问项目：对象代理数据库

这个就大概了解了下，我就具体介绍了下，说这是一个对象代理数据库的安卓项目，能让用户在手机上写SQL语句。不过面试官对这个数据库的具体内容不是很感兴趣

考编程：判断大小端存储

主要思想就是使用枚举,这样就可以”触碰”到机器的底层了,c0字段代表的是实际上机器地址的低位,因此我们只需要看打出来的是逻辑地址的高位0x12还是逻辑地址的低位0x78即可.

考编程:快速排序

面试官想让我说了一下思想,我刚开始说了一大堆解释快排的过程,最后面试官引导我说出了divide and conquer233,所以感觉还是要学会概括核心.

我用Python写了一个快排,主要就是实现了一个split

关于这个程序,我一开始本来想找有没有haskell，试图用它秒做,但是没有；不过其实用Python也可以,就用list comprehension就可以了.

之后面试官说我这个quicksort写出了冒泡排序的感觉,我感觉确实,然后说了也可以用双指针.然后还问了对于这个快速排序的影响他速度的因素是什么,回答是和数据有关,如果选取的那个数尽量接近中位数,效果会好.这里本来还想说一下各大<算法>中那个找中位数的神奇算法,不过怕自己讲不好就没有讲.

知识

面试官也问了一些关于技术栈的问题,但没有具体问,我也了解到了一些学习编译的重要资料,比如说源码,比如说目前流行的一些工具.总的来说收获也比较大.

感受

本次面试主要在具体知识层面考察.个人觉得面试中没有特别掉链子,但是有几处可以证明自己实力的机会没有把握住,有点可惜.不过因为面试官人特别好,感觉总体面试体验还不错.

第二轮(上级面)

这一轮应该是直接上级的面试,在这一轮的面试当中,主要是针对项目在问问题,同时对于知识的广度也有考察,但总的来说考察范围还是在编译这一方面. 基本上和第一轮面试没有什么重复的,感觉配合的确实很好.

问项目:为什么做lisp解释器,做的过程中的体会

我大概讲了在SICP做的lisp和我之后做的lisp的区别,然后提到了我希望做一些优化.

讲到这里面试官非常感兴趣,然后让我讲是哪方面的优化.我主要说了我对这个项目最不满意的地方在内存管理和垃圾回收,然后大概讲了引用计数法,对象探测法,以及复制清除法,最后还提到我觉得分代的思想应该比较有用.面试官在这里也非常excited, 和我介绍了一些其他的我所不知道的垃圾回收方法.

然后问到做这个的体会,我说了自己真实的体验,通过这个项目对底层的一些布局有了很好的理解,可以在以后研究gcc时派上用场,同时我也觉得还是需要更多地去研读一些文献,了解比较新的设计方法,不然只是重复造轮子.面试官表示很赞同,也介绍了他们所做的工作,提到了阿里在Open JDK方面做的贡献.

项目中遇到问题的解决? 我大概讲了我关于op设计方案改变的过程.

关于伯克利的经历

这里面试官大概询问了这次经历的性质,以及我在SICP中取得的A+成绩是什么概念.这里我也提到了是这门课让我走进了编译的大门.

最后还问了我这次经历给我的收获,我这里答了参加了很多lab,提高了动手能力.(我对这里不是很满意,我觉得最大的收获可能是学会了通过讨论和教别人学习)

关于mips CPU

面试官问了这个项目的一些基本信息(用什么语言写?是写到FPGA上吗?)然后问了我大概编写是怎么样一个流程.我回答里大概介绍了各个模块的作用,然后介绍下如何在软件调试成功后下板子(编译软核/写ram和rom的数据/写入代码)

项目中什么是最核心的?我回答了是有限自动机的设计,以及时钟上升下降沿逻辑逻辑要理清.

问知识

由于之前聊项目聊了很久,因此就简单地问了我几个问题,应该大部分是关于C语言的:

  • 进程与线程
  • 堆和栈的区别,各有什么好处
    这里我主要按照C语言的底层结构来答的.栈主要用来放AR,堆主要用来放动态分配的数据.栈需要编译器的支持,同时可以通过调用链来查询数据.而堆则需要维护指针,防止内存泄漏…etc
  • extern关键字
    这里我就说了我用过它来引入其他文件的全局变量.不过我在面试后两天读了C专家编程,了解到了这个问题其实想问的是指针和数组,感觉如果当时能够举个例子回答一下会比较加分.
  • 有无接触过多线程
    比较实诚地回答了只了解过一些概念,在golang中用过,但是没有写过实质性的程序. 后面面试官问了我锁的概念,可能想验证一下这个golang经历的真实性,我大概说了一下锁应用的场景.

总结

这次面试主要从项目层面考察,对知识的广度和深度的要求都更高了,也很关心底层的知识. 不过我觉得自己回答的还不错,可能是结合了项目的一些亲身体会.这次面试后,我也重新去审视了C语言,对C语言的底层有了更多的理解

第三轮（总监面）

总监面最大的体会就是：人外有人。我体会到了那种你会的别人都会，你不会的别人也会的绝望感。我非常感谢这一轮面试的面试官，我现在能够每天保持高强度的阅读量，全都是因为和他的面试。

我这里主要介绍一下面试官是如何对我“见招拆招”的

关于Mips -> 深入询问我关于体系结构的知识

  • 总结流水线的策略
  • RISC和CISC异同的分析
  • 超标量、多发射概念
  • 其他记不得了，总之不要以为自己会一些过时的CPU设计就沾沾自喜了，接下来对于ARM、RISC-V、Intel的处理器架构我都要去看一遍。

关于编译

  • 解释Yacc和Lex的作用
  • 请从编程语言设计的角度分析Python、C/C++、Java的区别：这道题真的把我问懵了，如果让我设计，我确实能够实现其中的各种特性，但是我确实回答不出这个问题。我觉得这是一个很好的问题，值得我之后一直去思考–如果我只是懂得如果写编程语言，那我和那些调参工程师有什么区别？
  • 进一步追问我关于JVM：我直接败下阵来，之后必须要补上这方面的知识

总结

很抱歉关于其他的一些问题，实在想不起来了，关于这次面试，我的感受如下：

无论是做研究还是工作，我们一定要明白：一方面，我们要在自己所用的领域潜的足够深；一方面，我们又要站的足够高，明白我们做一项工作的意义。我做到这一点之后，我觉得我会是一个优秀的工程师的。

(编译原理)一个极简语言的设计

post @ 2020-04-08

最近正在阅读虚拟机的设计，越发感觉到迫切地需要自己实现一个语言的虚拟机，在此基础上，我可以尝试不同的优化策略,下面是这个语言的一些基本特征

类型系统

  • 基本类型支持int，bool，char
  • 指针类型，其中保存的是变量的地址
  • 结构类型，可以在结构类型中使用基本类型作为属性，并为其分配实际的空间；同时支持结构属性，但是在此我们保存的是结构属性的指针。
  • 数组类型，在当前设计，我们只允许定义一维的定长数组，并且规定其类型只能是基本类型，指针类型，结构类型中的一个。

语言特性

  • 第一版不支持表达式运算
  • 可以定义函数，但不支持类似nonlocal的操作，函数对变量的查找除了形参和局部变量就是全局作用域。
  • 可以定义映射函数，使用memoization的方式进行操作，如果用户实际定义的函数并不“纯”，后果由用户承担
  • 支持if， while， for语句，支持嵌套的词法作用域
  • 支持赋值语句=，支持带类型推断的赋值语句:=

内置函数

  • 内存分配的函数malloc()和free()， 那同时也需要支持sizeof()
  • 支持go语言的make函数分配一个对象，

语言格局

  • 运行时栈
    • 在函数调用时通过栈帧的push和pop进行处理，格式(形参， 返回值， 保存寄存器， 父AR, 返回地址)
    • 同时词法作用域也会有自己的栈帧，格式
  • 常量区， 定义的全局变量、出现的字符串都存在常量区
  • 运行时堆

垃圾回收

  • 回收的对象是堆中分配的对象，包括用户malloc()的对象
  • 采用对象追踪法，不使用引用计数
  • 需要使用一个堆内存分配器，能够实现一些内存管理
    • 堆中应当使用一个合理的内存分配方法，以防止碎片化。
    • 堆中内存采用新旧两代的思想，每次分配内存总是尝试在旧内存进行分配
    • 可以在旧内存不够的时候，使用对象追踪法进行垃圾的回收，并将存活下来的对象分配到新内存，同时更新所有对象的引用
    • 堆中采用一套编址方法，能够使某个对象的旧地址依然能访问到新地址
    • 以上只是一些思路

待定设计：

  • 数组和结构是否自动放在堆中，还是需要显式地使用内存分配的函数来分配

实现顺序：

  • 1.实现内存分配器，实现以下API

    • 分配指定大小的内存
    • 向某个地址填入数据
    • 清空一块内存
    • 根据一个对象的地址，找到其最新地址

  • 2.完成前端及代码生成

    • 特别要完成对编译符号表的解析
    • 生成的代码，在运行时需要和符号表配合使用
  • 3.实现运行时栈和常量区
    • 栈需要配合编译的符号表使用
    • 常量区需要一个寻址方式
  • 4.完成底层虚拟机指令集的设计，并逐个实现
